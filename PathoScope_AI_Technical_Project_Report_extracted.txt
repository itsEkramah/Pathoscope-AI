SECTION 2 — ABSTRACT
Modern bioinformatics research is increasingly dependent on the rapid processing of Next-Generation Sequencing (NGS) data. Viral genomics, in particular, represents an essential vanguard for disease control, vaccine design, and evolutionary tracking. Traditional fragmented analysis strategies, which require bioinformaticians to execute multiple disparate command-line binaries, are severely bottlenecked by high startup latency, lack of reproducibility, and error-prone file integrations. To resolve these operational inefficiencies, we present PathoScope AI: a scientifically defensible, fully automated, end-to-end viral functional genomics pipeline developed using modular Python and unified software engineering paradigms.
PathoScope AI automates sequence ingestion, quality control, coordinate-aware six-frame open reading frame (ORF) predictions, similarity-based Swiss-Prot database alignments, conserved Pfam domain scanning via profile Hidden Markov Models (HMMs), and statistically corrected pathway enrichment analysis (incorporating Benjamini-Hochberg False Discovery Rate corrections). Finally, the pipeline integrates a zero-hallucination, retrieval-augmented AI interpretation layer that grounds clinical hypotheses exclusively in computed biological contexts and real-time PubMed literature citation indexing. The entire execution generates a MultiQC-style interactive HTML report and portable data files suitable for audit-ready scientific peer review.
SECTION 3 — INTRODUCTION
Functional genomics aims to understand the dynamic relationships between primary nucleotide sequences and phenotypic biological functions. In virology, where genomes are extremely compact and evolutionary pressure drives rapid mutagenesis, identifying coding boundaries, transcription coordinates, and homologous functional categories represents a primary bottleneck. Traditional sequence analysis relies heavily on manual file manipulation—copying sequence segments between BLAST web tools, piping stdout streams into custom parsing scripts, and manually plotting pathway enrichments. This manual approach is irreproducible, scales poorly to large metagenomic datasets, and is highly prone to human error.
PathoScope AI is designed around a scientifically correct pipeline philosophy which mandates that biological computation occurs first, statistical validation second, and AI-assisted biological interpretation last. In the era of Generative AI, bioinformatics faces a critical risk from AI hallucination: LLMs applied naively to genomic data tend to invent coding frames, fabricate p-values, and invent pathway hijacks. PathoScope AI addresses these risks by strictly separating deterministic bioinformatics computation from AI literature grounding. Drawing inspiration from modern reproducibility systems like nf-core and viral-ngs, the pipeline ensures configuration-driven, modular, and audit-ready executions designed to scale efficiently on local student laptop hardware.
SECTION 4 — PROJECT OBJECTIVES
To achieve a defensible, modular, and automated computational genomics portal, the system targets several core objectives:
Primary Pipeline Automation: To automate the end-to-end flow from raw sequence ingestion (supporting FASTA, FASTQ, and compressed files) to publication-grade results without human intervention.
Biologically Correct Algorithms: To implement biologically valid sequence quality trimming and six-frame ORF translations that resolve ribosomal overlaps, nested frames, and reverse complement constraints.
Evidence-Ranked Homology: To achieve high-performance sequence similarity searches using DIAMOND/BLAST against curated Swiss-Prot databases, augmented by local Smith-Waterman local alignment refinements.
Statistically Controlled Enrichment: To compute mathematically valid host pathway enrichments (KEGG/Reactome) using hypergeometric and ssGSEA models, fully corrected via Benjamini-Hochberg False Discovery Rate calculations.
Zero-Hallucination AI Grounding: To construct a retrieval-augmented AI interpretation layer that indexes PubMed abstract literature to explain host pathobiology and therapeutic markers without hallucination.
Open Science Reproducibility: To deliver an audit-ready reproducibility framework tracking runtime parameters, software versions, SQLite caches, command-line logs, and portable HTML reports.
SECTION 5 — SYSTEM ARCHITECTURE OVERVIEW
PathoScope AI is engineered as an orchestrator coordinates eight completely isolated, modular execution blocks. Each stage compiles its outputs to disk as standard files (GFF3, FASTA, CSV, JSON), serving as an auditable execution trace. The directory structure is highly standardized:

Modules communicate strictly via the filesystem. For instance, the Preprocessor parses the input file and writes a cleaned FASTA sequence; the ORF Predictor parses this sequence and outputs a coordinate GFF3 and a translated proteins FASTA; the Annotator reads this protein FASTA and queries DIAMOND to output an alignment table CSV. This completely decouples computation from presentation and guarantees that intermediate states are inspectable and reproducible, mimicking industrial nf-core workflows.
SECTION 6 — INPUT DATA HANDLING SYSTEM
Input quality determines the validity of all downstream genomics. PathoScope AI features a highly robust input ingestion engine capable of handling FASTA, FASTQ, and compressed files (e.g., .gz format via Python's standard `gzip` module). During ingestion, sequences undergo strict IUPAC nucleotide validation. Sequences are scanned to detect invalid symbols (anything outside the standard A, C, G, T, U and standard IUPAC degenerate bases R, Y, S, W, K, M, B, D, H, V, N). Corrupted records, blank sequences, and truncated headers are safely rejected.
If a compressed `.fastq.gz` sequence is uploaded, the app extracts suffixes dynamically, enabling the preprocessor stream loader to launch a gzipped text reader. If corrupted records or structural character errors bypass this layer, they propagate down the pipeline—leading to frame shifts in ORF prediction, artificial premature stop codons, and failed homology alignments. Preprocessing represents the primary defensive wall against computational failures.
SECTION 7 — PREPROCESSING MODULE
The Preprocessing Module manages raw sequence trimming and validation. It supports two distinct execution paths. First, it attempts to execute local command-line subprocess calls to FastQC and fastp if they are installed in the system environment. However, on student laptops or standard personal hardware, these tools are frequently missing. To address this, PathoScope AI is built with an advanced, Pure-Python High-Fidelity preprocessing fallback engine.
This Pure-Python engine replicates Illumina adapter clipping, sliding-window quality trimming (clipping the 3' end when average Phred scores across a 4bp window drop below 20), and sequence duplicate checking. It tracks ambiguous nucleotides (N) and safely discards any sequence where ambiguous bases exceed 5% or the length falls outside minimum/maximum boundaries (default 100bp to 500kb).
Technical Note: FASTA vs FASTQ inputs handle quality statistics fundamentally differently. FASTQ files represent raw read inputs with base-level PHRED quality strings (Q = ord(char) - 33), allowing per-base QC distributions. FASTA files represent pre-assembled, deterministic genomes without quality scores. PathoScope AI handles this seamlessly: if a FASTA is provided, per-base quality plots are safely skipped, and an educational warning explains the physical genomic difference to prevent user confusion.

SECTION 8 — ORF PREDICTION MODULE
ORF prediction is the most biologically critical phase of the pipeline. Naive ORF scanning—searching for open reading frames strictly from start-to-stop in a single reading frame—is biologically incorrect because it ignores the complexities of viral genomes: overlapping genes, transcription coordinates, strand biases, and nested frames. Viruses maximize their extremely compact genomes by utilizing multiple overlapping reading frames on both the positive and negative strands.
PathoScope AI implements a mathematically valid six-frame coordinate-aware ORF translator. The algorithm details include:
6-Frame Scanning: Reverse complement sequences are generated dynamically for negative-sense strand analysis (-1, -2, -3 frames).
Coordinate Tracking: Coordinates are tracked dynamically relative to the primary genomic sequence, normalizing for negative-strand indices.
Overlap Resolution: The algorithm resolves overlapping nested genes on the same reading frame by implementing a configurable overlap threshold (default 50bp). If two ORFs overlap on the same frame, the nested shorter frame is eliminated, preventing artificial coordinate inflation.
Peptide Translation: Only ORFs exceeding a biologically defensible size (default 30 amino acids) are translated using codon table maps, outputting a standard GFF3 coordinate table and a translated proteins FASTA.
SECTION 9 — FUNCTIONAL ANNOTATION MODULE
Functional annotation maps predicted translated peptides to known reference databases. Traditional string matching or longest common subsequence (LCS) algorithms are biologically incorrect because they ignore residue substitution penalties, hydrophobic similarity, insertions, and evolutionary gaps. Real bioinformatics annotation requires substitution matrices (such as BLOSUM62) and probabilistic alignment scores.
PathoScope AI executes local subprocess similarity searches using DIAMOND or BLASTp. DIAMOND is utilized as the default engine because it is up to 20,000 times faster than BLASTp while maintaining equivalent sensitivity on protein datasets. The module enforces strict confidence filters: hits are rejected if their Expectation value (E-value) exceeds 1e-5, query coverage is under 50%, or sequence identity is below 30%. To guarantee science-grade annotation, the top-ranked hit for each ORF undergoes local Smith-Waterman sequence alignment refinement in Python, and a weighted annotation confidence score is calculated (40% E-value weight, 30% sequence identity, 30% query coverage).
SECTION 10 — DOMAIN & PATHWAY MAPPING MODULE
Proteins represent functional molecules that operate in cellular networks. The Domain and Pathway Mapping module connects annotated viral homologs to conserved structural domains and host biochemical pathways. Structural domains are scanned independently using profile Hidden Markov Models (HMMs) against the Pfam database via local HMMER `hmmscan` subprocess execution.
Mapped annotations are dynamically mapped to KEGG orthology (KO) categories and Reactome pathways. Hardcoded or static pathway mappings are scientifically invalid because biological pathway databases undergo constant curation. PathoScope AI resolves this by integrating real-time KEGG REST API queries and Reactome pathway retrievals. To optimize computational latency and enable offline execution, all queried pathways are cached dynamically inside a local SQLite database (`pathways_cache.db`). If a query is duplicated, it is retrieved from the cache in milliseconds, saving bandwidth and preventing rate-limiting blocks.
SECTION 11 — STATISTICAL ANALYSIS MODULE
Pathway enrichment analysis evaluates whether host biochemical pathways are significantly overrepresented by the annotated viral homologs. Naive statistics can yield highly misleading biological claims: duplicate ORFs or redundant coding regions inflate query counts, and an incorrect background universe size distorts the statistical test's baseline probability.
PathoScope AI implements a rigorous, mathematically valid enrichment engine. Overrepresentation analysis (ORA) is conducted using the hypergeometric distribution test, which calculates the probability of drawing at least 'k' annotated genes belonging to a pathway from a genome size 'N' containing 'M' total pathway genes: P(X >= k) = 1 - sum_{i=0}^{k-1} ( (M choose i) * (N-M choose n-i) / (N choose n) ).
To correct for the multi-comparison problem (where testing hundreds of pathways simultaneously inflates the probability of false positives), the module calculates Benjamini-Hochberg False Discovery Rate (FDR) corrections. Only pathways passing an adjusted FDR q-value threshold of <= 0.05 are marked as statistically significant. Single-Sample Pathway Analysis (ssPA/ssGSEA) is also supported, ranking functional priorities by combining Swiss-Prot alignment scores and ORA p-values dynamically.
SECTION 12 — VISUALIZATION SYSTEM
Data visualization represents an essential component of supplementary scientific reports. PathoScope AI generates high-fidelity, journal-grade vector graphics. Preprocessing distributions (GC histograms, read retention waterfall charts) are plotted alongside genomic coordinate lane tracks mapping predicted ORF frames and strand orientation.
The Protein-Pathway Interaction (PPI) Bipartite Network map is compiled using networkx and Matplotlib styled over a stunning dark academic theme. Viral protein nodes are represented as glowing neon sky-blue circles (`#38bdf8`), while Host Target Pathways are represented as glowing neon violet diamonds (`#c084fc`) with sizes scaled proportionally to their connection degrees (hijack centrality). Semi-transparent edges (`#475569`, alpha=0.45) connect the nodes, visualizing host cellular hijack mechanics in an organic, intuitive manner. Static placeholder figures are avoided entirely, ensuring all plots represent real computed data.
SECTION 13 — AI INTERPRETATION SYSTEM
AI representation layers in bioinformatics are powerful but present severe biological risks. Standard LLMs behave as creative text engines: if asked to analyze a genomic file, they will fabricate non-existent p-values, hallucinate structural domains, and invent spurious pathologies. PathoScope AI resolves this by enforcing a strict, zero-hallucination Retrieval-Augmented Generation (RAG) architecture.
The AI module first retrieves real-time PubMed literature abstracts matching Swiss-Prot annotations and enriched pathway targets via the NCBI Entrez E-utilities API. It then builds a rigid system prompt injecting the deterministic bioinformatics context (actual GC%, predicted ORFs, alignment scores, significant FDR-corrected pathways) alongside the real PubMed abstracts. The model is restricted to interpreting this structured context, and must citation-track every claim using real PMIDs [PMID: XXXXXX].
Furthermore, if API endpoints are offline or quota-limited, the system seamlessly triggers a deterministic, offline rule-based fallback virology synthesizer. This fallback is dynamically grounded in Baltimore replication classes and host cellular pathologies, guaranteeing scientific rigor even in offline environments.
SECTION 14 — STREAMLIT DASHBOARD
To provide a premium user experience, PathoScope AI includes an interactive, high-performance Streamlit web dashboard. The UI matches modern dark slate and indigo aesthetics (`#0b0f19`). The panel allows users to upload standard genomics files or test the pipeline immediately using a Leviviridae Bacteriophage MS2 preset sequence. Slider controls enable real-time parameter adjustments—such as quality boundaries, minimum ORF lengths, DIAMOND alignment thresholds, and Benjamini-Hochberg FDR filters.
The dashboard displays a live runtime tracking status window showing execution times for each milestone. Completed results are presented across a comprehensive Results Explorer containing interactive tables, binned distributions, the dark-mode bipartite network map, citation bibliographies, and clear download buttons for portable HTML dashboard exports.
SECTION 15 — REPRODUCIBILITY FRAMEWORK
Reproducibility represents the core requirement of modern bioinformatics. PathoScope AI implements a strict, config-driven pipeline framework. All execution thresholds are configured externally using a structured YAML configuration file (`config/default_config.yaml`). The pipeline loads parameters model-checked via Pydantic (`config_loader.py`), ensuring that any invalid structural properties are rejected at startup.
Every completed run generates a structured `metadata.json` audit file in the output directory. This file logs: the input file's MD5 checksum (ensuring raw sequence integrity), execution status, operating system platform, Python environment details, exact CLI commands, individual stage runtime durations, and a complete snapshot of all configuration settings. This enables other researchers to duplicate the environment and reproduce the exact genomic results seamlessly.
SECTION 16 — SOFTWARE ENGINEERING DESIGN
The software architecture of PathoScope AI is engineered to modern industrial standards. The pipeline is coordinated by a unified `PipelineCoordinator` class which manages workspace directory setup, pipeline state transitions, temporary file clipping, and metadata auditing. Heavy bioinformatic import packages (such as BLAST parsers, HMMER connectors, and statistical models) are lazy-loaded dynamically inside their respective execution calls. This optimization minimizes app startup latency, reducing memory overhead on lightweight student hardware.
All subprocess calls are executed through a robust shell runner wrapper that captures standard output and error streams, managing retries and raising structured custom exceptions (e.g., `ToolExecutionError`, `DatabaseNotFoundError`) on failure. This defensive coding ensures that any module crash is handled gracefully, logging precise debugging traces without compromising the stability of the dashboard server.
SECTION 17 — TESTING & VALIDATION STRATEGY
Quality assurance is enforced through a comprehensive suite of **80 unit and integration tests** built using pytest. The testing strategy validates every computational module against realistic, boundary-condition datasets:
Preprocessor Tests: Verifies sequence validation under empty lines, invalid degenerate symbols, and duplicate headers.
ORF Translators: Validates overlapping reading frames, strand coordinate reversals, and nested ORF filtering on known phage sequences.
Alignment Realism: Ensures Smith-Waterman alignment scores and confidence calculations operate correctly.
Statistical Integrity: Verifies that ORA enrichment tests calculate correct p-values under varying universe bounds, and that Benjamini-Hochberg calculations match standard bioinformatics libraries.
Streamlit Dashboard Tests: Simulates user interactions, configuration mutations, slider settings, and initial rendering states.
SECTION 18 — DATASETS USED FOR TESTING
To validate pipeline performance on standard student laptops, several curated testing datasets are provided:
Bacteriophage MS2 Preset Sequence: A preloaded 613bp single-stranded RNA sequence mapping to Leviviridae MS2. Proves 6-frame ORF scanning, Pfam coat protein domain detection, and rule-based fallback interpretive summaries. Execution takes under 5 seconds.
Bacteriophage phiX174 Genome: A 5.3kb single-stranded DNA genome representing the microvirus phiX174. Used to test nested ORF overlap resolution (such as protein E nested within gene D) and Sanger sequence coordinate alignments. Execution takes under 15 seconds.
Simulated Metatranscriptomic Read Pool: A simulated metatranscriptomic list containing corrupted base structures, degenerate IUPAC bases, duplicate headers, and adapter sequence tags. Used to test preprocessor quality trimming filters and duplicate header renames.
SECTION 19 — RESULTS & EXPECTED OUTPUTS
A successful pipeline execution generates structured, auditable results under the versioned output directory. Inside `preprocessed/`, a cleaned sequence FASTA is written alongside the structured `qc_report.json` logging GC fraction and sequence duplication rates. Inside `orfs/`, a standard GFF3 file records coordinate intervals, reading frames, and parent strands for each gene. The `annotations/` directory outputs `annotated_proteins.csv`, which documents homology alignment details, Swiss-Prot identifiers, E-values, query coverages, and final annotation confidence scores.
The `pathways/` and `enrichment/` directories save tabular classifications and hypergeometric ORA results corrected via Benjamini-Hochberg calculations. Finally, `visualizations/` exports publication-ready graphics, and `final_report/` compiles the interactive HTML dashboard, providing researchers with complete supplementary files for academic publication.
SECTION 20 — LIMITATIONS
PathoScope AI represents a highly modular, student-accessible pipeline, but several computational and biological limitations remain. First, the local homology alignment relies on a curated Swiss-Prot reference database; novel metagenomic viral strains without close homologous representatives in public databases will map as hypothetical proteins. Second, Pfam domain searches and HMMER execution require substantial local memory; scanning massive, multi-megabase genomic assemblies can result in high computational latency on standard student hardware. Lastly, the AI interpretation layer is grounded in abstracts indexed by the PubMed API; complex structural protein conformations, dynamic binding kinetics, and in vitro experimental results remain unmodeled and require manual wet-lab validation.
SECTION 21 — FUTURE IMPROVEMENTS
To elevate PathoScope AI into a production-grade bioinformatics platform, several future upgrades are planned. First, we plan to migrate the core workflow management from Python-based shell runners to Nextflow. This will enable containerized, distributed execution on high-performance computing (HPC) clusters and cloud environments (such as AWS Batch). Second, we aim to integrate local deep learning structural models (like AlphaFold3 or ESMFold) to predict 3D structural folding coordinates for hypothetical proteins that fail standard database alignment. Finally, we will implement comprehensive metagenomic viral taxonomic binning tools (like Kraken2) to enable mixed-species profiling directly from environmental read samples.

The project demonstrates that robust, science-grade bioinformatics can run efficiently on student laptop hardware.
